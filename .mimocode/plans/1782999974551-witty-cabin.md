## Design Note: Future Action Memory Layer

### The Problem with Static layout_targets

Static `layout_targets` are a bootstrap mechanism, not the end state. They:
- Break when game UIs resize or update
- Require manual calibration per game per resolution
- Don't learn from successful actions
- Can't adapt to different screen layouts

### Where the Memory Layer Plugs In

The extension point is `locate_selector()` in `target_locator.py`. Currently the cascade is:

```
locate_selector(selector_key, profile, nodes, window_bounds)
  1. UIA match (auto_id/title from profile.selectors)
  2. action_mappings (title match)
  3. allow_coordinate_fallback → UIA substring match
  4. locate_layout_target (profile.layout_targets + window_bounds)
```

**The memory layer inserts between steps 3 and 4:**

```
locate_selector(selector_key, profile, nodes, window_bounds)
  1. UIA match
  2. action_mappings
  3. allow_coordinate_fallback → UIA substring match
  4. ACTION_MEMORY_LOOKUP (new) ← learned zones from past sessions
  5. locate_layout_target (static fallback, last resort)
```

### What Action Memory Stores

Per game_type + screen_state, persist:
```python
{
    "game_type": "uno",
    "screen_state": "in_game",  # from perception observation
    "action_zones": {
        "draw_button": {
            "x_ratio": 0.44, "y_ratio": 0.34,
            "confidence": 0.85,
            "success_count": 12,
            "fail_count": 1,
            "last_verified_ms": 1720000000000,
            "source": "learned",  # vs "profile" for static
        },
        "hand_card_0": { ... },
    }
}
```

### How Memory Updates

After each action execution:
1. `verify_screenshot_transition()` already runs → provides success/failure signal
2. On success: increase confidence, update timestamp
3. On failure: decrease confidence, optionally re-sample coordinates from screenshot perception
4. After N consecutive failures: mark zone as stale, trigger re-detection

### How Memory Feeds Back into Target Location

The `HeuristicCanvasUNOPlugin` already detects regions in screenshots. The memory layer connects that perception to action grounding:

1. Screenshot perception detects `ScreenRegion(region_id="draw_pile", x=844, y=367, ...)`
2. Memory layer maps `selector_key="draw"` → `region_id="draw_pile"` (learned association)
3. Target locator uses the detected region's coordinates instead of static layout_targets

### What Replaces Static layout_targets

Static `layout_targets` become the **cold-start fallback** — used only when no learned memory exists for a game. As the agent plays more sessions:

1. First session: uses layout_targets (current state)
2. Successful actions → memory records zones with confidence
3. Subsequent sessions: memory zones take priority over layout_targets
4. layout_targets remain as fallback for truly new games

The profile JSON keeps `layout_targets` as a bootstrap seed. The runtime memory store supersedes it once enough successful actions have been recorded.
